# Open Report — Cloud development workflow

This document describes the dual-repo workflow used to maintain Open Report
as both an open-source project (AGPL-3.0) and a hosted commercial offering.

It is meant for the project maintainer. End users of either edition can ignore it.

> **License note.** The OSS edition is licensed under AGPL-3.0. The cloud
> edition is a derivative work that is dual-licensed by the copyright holder
> under a proprietary license. This dual-licensing is only valid as long as
> the copyright holder owns rights to all the code merged into the cloud
> repository. Future external contributors must therefore sign a CLA before
> their code is accepted upstream.

## Repository layout

```
github.com/cracky777/openreport          (public, AGPL-3.0)   ← OSS edition
github.com/cracky777/openreport-cloud    (private)            ← Cloud edition
```

The **OSS repo** is the single source of truth for everything that ships in the
open-source edition: the core engine, widgets, model editor, RLS, etc.

The **cloud repo** starts as a clone of the OSS and adds:

- Billing / subscriptions
- Multi-tenant isolation primitives
- Premium features (e.g. SSO, audit logs, scheduled exports)
- Cloud-only middleware and infrastructure code

The cloud repo **never modifies OSS files** if it can be avoided — it only
adds new files in dedicated `cloud/` directories.

## Initial setup (one-time)

These steps assume you've already created the **private** `openreport-cloud`
repository on GitHub (empty — no README/LICENSE/.gitignore generated by GitHub).

```bash
# Move next to the existing OSS clone
cd c:/Users/JonathanFritsch

# Bootstrap the cloud repo from the OSS one
git clone https://github.com/cracky777/openreport.git open-report-cloud
cd open-report-cloud

# Rename the OSS remote to "upstream", then add the private remote as "origin"
git remote rename origin upstream
git remote add origin https://github.com/cracky777/openreport-cloud.git

# Initial push to the private repo
git push -u origin master
```

You now have two local clones:

| Local folder              | Remote `origin`                                              | Remote `upstream` |
|---------------------------|--------------------------------------------------------------|-------------------|
| `open-report/`            | `cracky777/openreport` (public)                              | —                 |
| `open-report-cloud/`      | `cracky777/openreport-cloud` (private)                       | `cracky777/openreport` (public) |

Verify with `git remote -v` in each folder.

## Daily workflow

### a) Adding a feature to the OSS edition

Work in the **OSS clone** (`open-report/`) exactly like you do today.

```bash
cd c:/Users/JonathanFritsch/open-report
# code, test, commit
git push                         # → publishes to the public repo
```

### b) Pulling OSS changes into the cloud edition

Whenever you push OSS work, sync it into the cloud repo. **Do this often** —
small, frequent merges produce fewer conflicts than monthly catch-ups.

```bash
cd c:/Users/JonathanFritsch/open-report-cloud
git fetch upstream
git merge upstream/master        # merges OSS work onto your cloud branch
# resolve conflicts if any (rare if you respect the folder conventions)
git push                         # → publishes to the private repo
```

If you're confident the cloud repo has no diverging commits and just want
the OSS state, you can fast-forward instead:

```bash
git pull --ff-only upstream master
```

### c) Adding a cloud-only feature

Work in the **cloud clone** (`open-report-cloud/`) and stay inside the
`cloud/` directories so OSS files remain untouched.

```bash
cd c:/Users/JonathanFritsch/open-report-cloud
# add files under server/cloud/ or client/src/cloud/
git add server/cloud/routes/billing.js
git commit -m "Add Stripe billing webhook"
git push                         # → publishes to the private repo only
```

⚠️ Cloud features must **never** be pushed to the OSS repo. Always check
your remotes (`git remote -v`) before a push.

### d) Promoting a cloud feature to OSS

Sometimes a feature you started as cloud-only turns out to be a good fit
for the OSS edition. To promote it:

1. Move the relevant files out of `cloud/` and into the regular tree
   (e.g. `server/cloud/routes/foo.js` → `server/routes/foo.js`).
2. Commit the move in the **cloud repo** to make sure the local state is clean.
3. Cherry-pick the relevant commit(s) onto the OSS clone:
   ```bash
   cd c:/Users/JonathanFritsch/open-report
   git remote add cloud /c/Users/JonathanFritsch/open-report-cloud   # one-time
   git fetch cloud
   git cherry-pick <commit-sha>
   git push
   ```
4. Re-sync the cloud repo to remove the duplicate (the upstream merge will
   bring the OSS version in, then drop the now-redundant cloud copy).

## Folder conventions

Anything cloud-only lives under a `cloud/` directory:

```
server/
  routes/                        ← OSS
  middleware/                    ← OSS
  cloud/                         ← cloud-only (private repo only)
    routes/
      billing.js
      subscriptions.js
    middleware/
      tenantIsolation.js
      licenseCheck.js
    index.js                     ← entry point — registers everything

client/
  src/
    pages/                       ← OSS
    components/                  ← OSS
    cloud/                       ← cloud-only (private repo only)
      pages/
        Account.jsx
        Billing.jsx
        Plans.jsx
      components/
      index.js
```

This makes diffs trivial to review and keeps the merge boundary obvious:
**nothing in `cloud/` ever leaves the private repo.**

## Conditional activation

Cloud code is loaded only when the build is configured for cloud.
Use the `OPENREPORT_CLOUD` environment variable as the gate.

### Server (`server/index.js`)

```js
// Cloud-only routes / middleware. Loaded only when explicitly enabled so the
// public OSS build never depends on the private modules.
if (process.env.OPENREPORT_CLOUD === '1') {
  try {
    require('./cloud').register(app);
  } catch (err) {
    console.error('[cloud] Failed to load cloud module:', err.message);
    process.exit(1);                           // fail fast in production
  }
}
```

### Client (`client/src/App.jsx`)

```jsx
// Conditional dynamic import keeps the OSS bundle free of cloud code.
const isCloud = import.meta.env.VITE_OPENREPORT_CLOUD === '1';
const CloudRoutes = isCloud
  ? React.lazy(() => import('./cloud'))
  : null;

// ... inside the Routes element:
{CloudRoutes && (
  <Suspense fallback={null}>
    <Route path="/cloud/*" element={<CloudRoutes />} />
  </Suspense>
)}
```

In production (Docker, Render, Fly, etc.) the cloud build sets the variables;
the OSS build leaves them unset.

### `.env` files

```
# .env (cloud build)
OPENREPORT_CLOUD=1
VITE_OPENREPORT_CLOUD=1
STRIPE_SECRET_KEY=...
```

## Conflict-prevention rules

Most merge conflicts happen when an OSS file is edited by both repos.
Apply these rules to keep merges painless:

1. **Cloud changes go in `cloud/` directories.** Period.
2. If a cloud feature *needs* the OSS to expose something, **add the
   extension point in the OSS first** (a hook, a provider, a config
   field). Push it to OSS, sync into cloud, then implement the cloud
   side using the new extension point. The OSS edition benefits too.
3. When a tiny OSS edit is unavoidable (e.g. mounting a `<Router>` slot),
   keep it minimal and tag it: `// CLOUD-HOOK: ...`. This makes the
   modification easy to spot during merges.
4. **Sync from upstream often** (ideally at the start of every cloud
   work session). The longer the lag, the more conflicts pile up.
5. Never push OSS-tracked files from the cloud repo to a public branch.
   Configure `origin` in the cloud clone to point only to the private repo.

## Pre-push safety checks

To avoid leaking cloud code into the OSS by accident, add a pre-push hook
to the cloud clone (`open-report-cloud/.git/hooks/pre-push`):

```bash
#!/usr/bin/env bash
# Block pushes from the cloud clone to the OSS remote.
remote_url="$2"
if echo "$remote_url" | grep -q 'cracky777/openreport\.git'; then
  echo "ERROR: refusing to push from the cloud clone to the public OSS repo."
  echo "  Did you mean 'origin' (private)? remote was: $remote_url"
  exit 1
fi
```

Then `chmod +x .git/hooks/pre-push` (Linux/Mac) or just save the file (Windows
will execute it via Git Bash).

## Building & deploying the cloud edition

Tag both repos with matching versions when you ship a release. The cloud
edition's CI builds a Docker image with `OPENREPORT_CLOUD=1` and the OSS
secrets baked in (Stripe keys, etc.):

```dockerfile
# Dockerfile.cloud (lives in the private repo only)
FROM node:20-alpine AS build
WORKDIR /app
COPY . .
RUN npm ci && npm --prefix client run build
ENV OPENREPORT_CLOUD=1
ENV VITE_OPENREPORT_CLOUD=1
CMD ["node", "server/index.js"]
```

The OSS edition keeps using the existing top-level `Dockerfile` — unchanged.

## Troubleshooting

**"I pushed cloud code to the public OSS repo by mistake."**
1. Identify the bad commit(s): `git log --oneline`.
2. Force-push the previous good state:
   ```bash
   git reset --hard <good-sha>
   git push --force-with-lease origin master
   ```
3. Rotate any secret that was exposed (Stripe keys, etc.) — assume the
   commit was indexed within seconds of the push.

**"Merge conflicts on every upstream pull."**
You're probably modifying OSS files from the cloud repo. Find where the
edits live (`git log upstream/master..HEAD -- <oss-file>`), refactor them
into proper extension points in the OSS, and replay your work as additions.

**"The cloud build runs locally but the OSS build crashes."**
You probably referenced a cloud-only module from an OSS file. Search for
`require('./cloud')` or `import('./cloud')` outside of the
`OPENREPORT_CLOUD === '1'` guard and remove it.

## Checklist before each cloud release

- [ ] OSS `master` is green (CI passes).
- [ ] Cloud repo has no uncommitted changes.
- [ ] `git fetch upstream && git merge upstream/master` is clean.
- [ ] Cloud-specific tests pass (when they exist).
- [ ] The OSS edition still builds without `OPENREPORT_CLOUD=1`.
